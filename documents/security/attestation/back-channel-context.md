# Back-Channel Context

## Purpose

Marshaller parameter structs carry `std::vector<rpc::back_channel_entry>` in
both input and output paths. These records are intended for public metadata
about a message, not for encrypted application payloads.

Examples:

- attestation context references;
- route proof references;
- key ids;
- protected-envelope version/capability markers;
- OpenTelemetry ids;
- diagnostic correlation ids.

## Type Ids

Back-channel `type_id` values should use the existing IDL/type fingerprint
mechanism, for example `rpc::id<T>::get(version)`, rather than a separate
manually maintained registry.

The payload is a serialized blob of that type using the message's negotiated
encoding.

The fingerprint is version-aware. Two incompatible versions of the same context
type must produce different `type_id` values, which is what the IDL
fingerprint mechanism is intended to provide. A 64-bit collision is expected to
be vanishingly unlikely; if a payload cannot be parsed as the advertised type,
the receiver should treat it as malformed or hostile context rather than trying
to guess another schema.

Candidate context types:

```text
attestation_context
attestation_backend_context
attestation_verdict_context
protected_envelope_context
route_context
telemetry_context
security_failure_context
```

The exact IDL definitions are deferred.

## Public Metadata

Back-channel entries are readable by intermediates. Do not put encrypted RPC
payloads, plaintext method arguments, session keys, private keys, or other
secret material in them.

If a context item must be private to the endpoint, include it inside the
protected envelope plaintext instead.

## Mutation Model

Back-channel vectors are public and mutable while a message is routed.
Intermediates may append entries, for example OpenTelemetry correlation data or
route diagnostics. Protected RPC must therefore treat the received outer vector
as authoritative public context, not as data that must match the sender's
encrypted payload or AEAD associated data.

Future work may distinguish:

- source-signed immutable public context;
- hop-appended public context;
- diagnostic context generated by the receiving service.

If intermediates append security-relevant context, those appended entries need
their own integrity protection and issuer identity.

## Capability Negotiation

Protected-envelope support can be advertised through a back-channel capability
entry or through the reserved attestation/control interface. If policy requires
protected envelopes and the peer does not advertise support, return
`ZONE_NOT_SUPPORTED`.

`try_cast` remains the existing remote interface discovery mechanism after a
session exists or when policy explicitly permits it. Bootstrap does not require
pre-attestation `try_cast`: a caller may speculatively invoke the reserved
attestation/control function by object id and interface fingerprint, then handle
`ZONE_NOT_SUPPORTED` or another minimal failure if the peer does not implement
it.

# Nanopb Protocol Buffers Implementation Notes

Status note: this is a historical implementation plan. Many of the items below
are now implemented, including `rpc::encoding::nanopb`, generated
`nanopb_serialise` / `nanopb_deserialise` methods, CMake backend selection, and
the generic `to_nanopb` / `from_nanopb` serializer helpers. For current
user-facing behavior, see [Nanopb](nanopb.md); for source-of-truth behavior, see
`cmake/CanopyGenerate.cmake`, `generator/src/nanopb_generator.cpp`, and
`c++/rpc/include/rpc/internal/serialiser.h`.

This document records design notes for the Nanopb-backed Protocol Buffers path
in Canopy. The implementation goal is C++ serialization/deserialization
compatibility with standard Protocol Buffers wire format while avoiding the
Google C++ protobuf runtime in SGX enclaves.

## Goals

- Keep `.proto` files as the canonical compatibility contract.
- Preserve interoperability with other Protocol Buffers implementations.
- Provide C++ serialization/deserialization through Canopy generated code.
- Avoid reflection, descriptors, text format, JSON mapping, dynamic messages, gRPC, and generated Google C++ message APIs.
- Avoid linking enclave code against any protobuf runtime that was not compiled with SGX enclave flags.
- Avoid closed-source and copyleft dependencies in the protobuf runtime path.

## Non-Goals

- Do not implement gRPC.
- Do not expose Nanopb generated C structs as the public Canopy C++ API.
- Do not require reflection or descriptor metadata at runtime.
- Do not remove the existing Google C++ protobuf backend until Nanopb compatibility is proven.
- Do not support every protobuf feature initially; implement the subset generated by Canopy IDL first.

## Dependency

Nanopb is available as a submodule at:

```text
c++/submodules/nanopb
```

The expected runtime files are:

```text
c++/submodules/nanopb/pb_common.c
c++/submodules/nanopb/pb_decode.c
c++/submodules/nanopb/pb_encode.c
c++/submodules/nanopb/pb.h
c++/submodules/nanopb/pb_decode.h
c++/submodules/nanopb/pb_encode.h
```

The expected generator entry point is:

```text
c++/submodules/nanopb/generator/nanopb_generator.py
```

Nanopb uses the zlib license, which is permissive and non-copyleft. The license should still be included in release
license documentation when this backend becomes a shipped dependency.

## High-Level Design

Canopy will continue generating `.proto` schema files from IDL. The Nanopb backend will add a second generated C++ path
that uses Nanopb-generated C code for protobuf wire encoding and decoding.

Host builds may build both protobuf implementations:

- Google C++ protobuf for the current `to_protobuf` and `from_protobuf` path.
- Nanopb for new `to_nanopb` and `from_nanopb` compatibility tests.

SGX enclave builds should use only the Nanopb path:

- compile Nanopb runtime C files with SGX enclave compile flags,
- compile Nanopb-generated `.pb.c` files with SGX enclave compile flags,
- compile Canopy-generated Nanopb C++ adapters with SGX enclave compile flags,
- do not compile Google-generated `.pb.cc` files,
- do not link `protobuf::libprotobuf`, `libsgx_protobuf.a`, Abseil, gRPC, or other Google protobuf runtime libraries.

## Generated API

The new generated C++ methods should be separate from the current Google protobuf methods while the backend is being
proved:

```cpp
void nanopb_serialise(std::vector<char>& buffer) const;
void nanopb_deserialise(const std::vector<char>& buffer);
```

The generic serializer helpers in `c++/rpc/include/rpc/internal/serialiser.h` should add:

```cpp
template<typename T> uint64_t nanopb_saved_size(const T& obj);
template<class OutputBlob = std::vector<std::uint8_t>, typename T> OutputBlob to_nanopb(const T& obj);
template<typename T> std::string from_nanopb(const rpc::byte_span& data, T& obj);
```

These functions should call the generated `nanopb_serialise` and `nanopb_deserialise` members, mirroring the existing
`to_protobuf` and `from_protobuf` shape.

SGX builds should use `rpc::encoding::nanopb` for protobuf-compatible wire bytes. Host builds can continue using the
full protobuf backend where the Google C++ runtime is wanted, or use Nanopb where a smaller dependency surface is more
important.

## Generator Work

Add a new generator component named `nanopb_generator`.

It should reuse common concepts from the existing protobuf generator where practical:

- protobuf type mapping,
- sanitized field names,
- package and message naming,
- interface request/response message naming,
- struct traversal,
- template instantiation handling,
- enum handling.

The implementation should not copy Google C++ message API calls such as:

```cpp
SerializeToArray
ParseFromArray
ByteSizeLong
mutable_*
set_*
```

Instead, generated Canopy adapter code should:

- populate Nanopb C structs,
- configure callbacks for variable-length fields,
- call `pb_encode`,
- call `pb_decode`,
- translate errors into Canopy error paths.

## Runtime Support Layer

Add support headers under:

```text
c++/rpc/include/rpc/serialization/nanopb
```

This layer should hide repetitive Nanopb callback setup from generated code. It should provide helpers for:

- `std::string`,
- `std::vector<uint8_t>` and byte-like vectors,
- repeated scalar fields,
- nested message fields,
- map fields where Canopy-generated schemas require them,
- error handling and buffer stream setup.

The support layer must not perform file IO, network IO, logging that depends on OS facilities, descriptor lookup, or
dynamic reflection.

## CMake Work

Add CMake support in stages:

1. Locate the Nanopb submodule and expose runtime source paths.
2. Add a host Nanopb runtime target for tests.
3. Add an enclave Nanopb runtime target compiled with `CANOPY_ENCLAVE_COMPILE_OPTIONS` and `CANOPY_ENCLAVE_DEFINES`.
4. Extend proto generation so selected IDL targets can invoke the Nanopb generator.
5. Add Nanopb-generated `.pb.c` files to the relevant generated target.
6. For enclave IDL targets, select Nanopb-generated sources instead of Google `.pb.cc` sources.

The final SGX link line must not contain Google protobuf or gRPC libraries.

## Compatibility Tests

Add tests in `c++/tests/serialiser_test` while both backends are available:

- `to_nanopb` then `from_nanopb`,
- `to_protobuf` then `from_nanopb`,
- `to_nanopb` then `from_protobuf`.

Start with:

- scalar struct fields,
- strings,
- empty strings,
- large strings,
- nested structs,
- vectors,
- maps,
- template-generated structs,
- enums where available.

The cross-backend tests prove wire compatibility. The direct Nanopb round-trip proves the new generated adapter path.

## SGX Verification

After host compatibility tests pass:

1. Configure `Debug_SGX_Sim` with protocol buffers enabled.
2. Build the generated IDL enclave targets that use protobuf.
3. Build the smallest enclave transport and marshal test targets.
4. Inspect the failing or final link line and confirm no Google protobuf runtime is linked.
5. Build `marshal_test_enclave`.

If a Nanopb runtime function is missing or unsuitable for SGX, fix it by changing how the runtime target is compiled or
by narrowing generated schema support. Do not work around the issue by linking non-enclave protobuf libraries.

## Implementation Order

1. Add `nanopb_generator` skeleton and wire it into the generator build.
2. Add `to_nanopb`, `from_nanopb`, and `nanopb_saved_size`.
3. Add basic runtime support headers.
4. Generate and compile Nanopb sources for one serialiser test IDL.
5. Implement scalar and string encode/decode.
6. Add cross-backend tests for scalar/string structs.
7. Extend to nested structs, vectors, maps, templates, and enums.
8. Switch enclave protobuf source selection to Nanopb.
9. Verify SGX simulation build.

## Main Risks

- Nanopb represents variable-length fields through callbacks or configured fixed limits; Canopy must choose callback-based
  handling where dynamic C++ containers are needed.
- Protobuf maps are generated as repeated entry messages; the adapter needs explicit map support.
- Cross-IDL imports must produce consistent Nanopb names and include paths.
- SGX builds must compile all Nanopb runtime and generated C files with trusted compile flags.
- The existing protobuf generator has some host-only assumptions; shared logic should be extracted carefully rather than
  changing host behavior during the first Nanopb pass.
